# FULLSTP — Atomic System Analysis

Date: 2026-04-08

## 1. TRUE PURPOSE (FIRST PRINCIPLES)

Input: A business name + interview answers (brand mood, audience, value prop) Output: A fully deployed, content-seeded website on a live domain Core Transformation: Natural language business description → structured BMC → 5-agent AI content package → provisioned Next.js + Payload CMS tenant on a remote server Constraints: One Claude API call chain (~5 sequential LLM calls), SSH availability, HestiaCP server reachability, Unsplash image API, ~5–10 min end-to-end build time

This system takes a business description from a human, interviews them with AI to extract a Business Model Canvas, and uses a 5-agent swarm to build and deploy a full website — no human in the loop between intake and launch.

## 2. SYSTEM LAYER MAPPING

| Layer | Components |
| --- | --- |
| Interface | `src/app/(site)/page.tsx` → MultiPhaseChat (React chat UI, 5 phases: landing → strategy → auth → building → operational). SSE streams from `/api/swarm`. |
| Orchestration | `/api/swarm/route.ts` — 3 modes: conversation, pipeline, operations. `SwarmPipeline.run()` — 9-stage sequential build. `activeBuildDomains` Set — deduplication guard. |
| Cognition | QueenAgent (Claude Sonnet, strategy + consensus), DesignDirectorWorker (Claude Haiku, visual selection), ContentWriterWorker (Claude Sonnet, copy), UIArchitectWorker (Claude Sonnet, layout), PayloadExpertWorker (Claude Sonnet, block mapping). All prompt templates live in `prompts.ts`. |
| Tool | `deployTenantViaBridge()` (SSH → HestiaCP server), `fetchUnsplashImages()` (stock photos), `payload.create/update()` (Local API DB writes), `bootstrap-tenant.ts` (remote tenant seeding), `preset-loader.ts` (JSON template hydration). |
| State | SQLite (`data/database.db`): customers (phase lifecycle), bmcs (raw business canvas), deployments (domain, port, creds, status), pages (factory-side). SharedMemory (in-process Map, scoped to one pipeline run). Preset JSON files (`src/lib/swarm/presets/*.json`) as static structural state. |

## 5. CONTROL POINT IDENTIFICATION

| Control Point | Why It's a Leverage Point |
| --- | --- |
| `CEO_CHAT_SYSTEM` prompt | The entire BMC (which drives ALL downstream output) is shaped by how this prompt guides the interview. Change the questions, change every tenant site ever built. |
| `QUEEN_STRATEGY_SYSTEM` prompt | `businessArchetype` selection here locks in page slugs, nav structure, and preset routing. Wrong archetype = wrong site structure for all 9 downstream stages. |
| `pagePresets` field in DesignBrief | The fork between "preset path" (fast, deterministic) and "legacy path" (slow, LLM-generated structure, hallucination-prone) lives entirely here. If this field is absent, 3 extra Claude calls trigger including the consensus loop. |
| Preset JSON files (`/presets/*.json`) | These are the actual HTML structure of every site. No LLM can hallucinate block field names because structure is fixed. Every new block type or layout change requires editing these files. |
| SharedMemory instance | The only mechanism for agent-to-agent context passing within a build. If it's lost (process crash), the pipeline falls back to `fallbackContentGeneration()`. Its content dictates what each downstream agent receives. |
| `activeBuildDomains` Set | The only concurrency guard. In-process only — a server restart clears it, potentially allowing duplicate tenant builds. |
| `DEPLOY_SSH_KEY` / `DEPLOY_SERVER_IP` env vars | These determine whether a real deployment happens or a "simulated" one. In simulated mode, no live site is created but state is still persisted. This is the system's entire deployment on/off switch. |

Branching: At `pagePresets` check — cuts 3 LLM calls and the consensus loop. Loop: Consensus validation in legacy path — max 2 healing attempts before forced exit. Context degradation: SharedMemory is in-process per `pipeline.run()` call. Each agent only sees what prior agents explicitly `set()`. No implicit cross-agent memory.

## 7. HIDDEN ASSUMPTIONS

| Assumption | Where It's Implicit |
| --- | --- |
| Claude returns valid parseable JSON | Every agent call uses `parseJSON<T>()`. On failure, entire swarm throws → fallback triggered. No retry on malformed response. |
| `pagePresets` field is always populated by DesignDirector | The fast path assumes Haiku reliably includes this. If Haiku omits it, the expensive legacy path runs silently. |
| Remote server at `167.86.81.161` is reachable and HestiaCP is configured | Hardcoded IP. No DNS fallback. No health check before pipeline begins. |
| SSH key or password in env vars is valid | `isDeploymentConfigured()` only checks if the vars exist, not if the connection works. Discovery happens mid-pipeline. |
| Golden image source in `src/golden-image/` is always buildable | The local validation checks file presence but not TypeScript compilation. A broken golden image = all tenant deployments fail. |
| Unsplash API is available (non-blocking) | System continues without images. CSS fallbacks assumed in golden image. |
| Preset JSON structure matches what the golden image's CMS expects | No runtime schema validation between factory's preset files and the deployed tenant's Payload block configs. Divergence = silent CMS errors on tenant. |
| SharedMemory survives the entire pipeline run | It's a plain in-process Map. Any unhandled rejection kills it. The try/catch in `run()` catches this and falls back. |
| Port allocation is collision-free | `getNextPort()` / `getUsedPorts()` reads from the SQLite Deployments table. If DB is corrupt or two builds run simultaneously on different processes, port collision is possible. |
| Tenant's `bootstrap-tenant.ts` stdout is JSON parseable | The bridge parses this to extract `adminEmail`, `adminPassword`. Parse failure = credentials lost, customer stuck in building phase forever. |

## 8. FAILURE ANALYSIS

LLM produces malformed JSON (`e.g. QueenAgent.generateStrategy`): `parseJSON<StrategyBrief>()` throws → `swarmContentGeneration()` catches → `trackedLog('Factory', 'Swarm error: ..., 'error')` → `fallbackContentGeneration(bmc, log)` runs → static `ARCHETYPE_CONFIGS` templates used → pipeline continues with zero LLM-generated content → deployment proceeds with generic template text. User gets a live site with placeholder copy.

SSH / HestiaCP fails mid-deploy: `deployTenantViaBridge()` throws or returns `{ success: false }` → pipeline persists a Deployments record with `status: 'error'` → `emit('build_error', { error })` → SSE stream closes. Customer stays in building phase. No automatic retry. Manual re-run required. The domain may be partially registered on HestiaCP.

SharedMemory lost (process crash): The try/catch in `SwarmPipeline.run()` wraps only `swarmContentGeneration()`. If the crash happens during deploy (after content generation), state is partially persisted in SQLite. If before, `fallbackContentGeneration()` runs. Full process crash → SSE stream disconnects → client shows error. No resume capability.

Consensus loop never converges (legacy path only): `healAttempts > MAX_HEAL_ATTEMPTS (2)` → `log('Queen', 'Max healing attempts reached...', 'error')` → pipeline proceeds with the last generated `finalContent` regardless of alignment. Deploy proceeds with potentially malformed block data → some blocks may fail to render in the tenant CMS.

Latency spike (Claude API slow): Each Claude call is `await`-ed. No timeout set on any Anthropic SDK call. A 60+ second Claude response stalls the SSE stream. Browser may show the last log entry indefinitely. No keepalive pings on the SSE stream itself. Client may timeout and lose connection while the server-side pipeline continues running in the background. Domain gets deployed but client never receives `build_complete`.

Port collision (two simultaneous builds): `getUsedPorts()` reads the Deployments SQLite table. Two concurrent `pipeline.run()` calls in the same process read the same used ports list before either writes → same port assigned to both tenants → second deploy fails with `port-in-use` error. The `activeBuildDomains` Set prevents the same domain twice but not two different domains racing on port allocation.

## 9. MINIMAL REBUILD (FIRST PRINCIPLES RECONSTRUCTION)

To rebuild the core loop from scratch with the smallest viable system:

### Step 1: Single interview endpoint

One POST route. Accepts `{ messages }`. Calls Claude with a system prompt that asks business questions. Returns either a message or `{ _strategyComplete: true, ...bmc }`. No state stored yet.

### Step 2: Content generation function

One function: `generateContent(bmc) → ContentPackage`. Call Claude once with a prompt that combines strategy extraction + content writing. Output a flat JSON of pages → blocks → text fields. No multi-agent architecture needed for MVP.

### Step 3: Static preset templates

Replace step 2's LLM structure generation with JSON preset files. LLM only fills in `{{placeholder}}` tokens. Eliminates hallucination of block field names entirely.

### Step 4: Tenant deployment function

One function: `deployTenant(domain, port, contentPackage)`.

- SSH to server
- Copy golden image source
- Write `.env` with domain/port/secret
- Run `bootstrap-tenant.ts` (seeds DB)
- `pm2 start` the process
- Register domain in HestiaCP

### Step 5: State persistence

Three SQLite tables: customers (name, email, phase), bmcs (json blob), deployments (domain, port, status, adminEmail, adminPassword). Payload CMS handles this already.

### Step 6: Real-time feedback

Wrap steps 2–5 in an SSE ReadableStream. Log each step with `{ agent, text, status }`. Send `build_complete` with credentials at end.

Minimal viable architecture:

```text
POST /api/swarm → SSE stream
  ├─ mode=conversation → Claude(CEO_SYSTEM + messages) → question or BMC
  └─ mode=pipeline →
       generateContent(bmc)     [1 Claude call, preset template fill]
       deployTenant(domain, pkg) [SSH + PM2 + HestiaCP]
       persistState()            [SQLite via Payload Local API]
       emit('build_complete', creds)
```

That's 1 route, 1 LLM call for content, 1 deploy function. The current system's 5 agents exist to improve quality and specialization of output — they're not architectural requirements.

## 10. FINAL COMPRESSION

This system takes a business description (entered via a conversational UI), routes it through a multi-turn Claude interview to extract a Business Model Canvas, then launches a 5-agent AI swarm (Queen → Design Director → Content Writer → UI Architect/Preset Loader → Payload Expert) that produces a complete Payload CMS content package, which is then SSH-deployed as a pre-built Next.js golden image onto a HestiaCP-managed VPS, seeded with all generated content, assigned a domain with TLS, and handed back to the operator as a live website — all within a single SSE-streamed pipeline call, producing a deployed business site from a conversation in under 10 minutes.
